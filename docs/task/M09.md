# M09 — Test di integrazione convergenza cluster

## Stato iniziale

Prima di questa milestone il repository disponeva di test di convergenza e robustezza principalmente nei package interni, in particolare in `internal/gossip` e nei package delle aggregazioni. Mancava però una suite di integrazione esplicita in `tests/integration/` con un entrypoint canonico dedicato alla verifica della convergenza del cluster come comando operativo esterno ai package interni.

## Gap

I gap principali rispetto al task M09 erano i seguenti:

- assenza della directory canonica `tests/integration/`;
- assenza di un test chiamato esplicitamente `TestClusterConvergence`;
- assenza di un documento canonico `docs/testing.md` che congelasse criteri, timeout e comando ufficiale;
- README non ancora aggiornato per dichiarare formalmente la disponibilità del test M09 e il relativo comando operativo.

## Modifiche

Le modifiche introdotte in questa milestone sono:

1. mantenimento della directory `tests/integration/` come sede canonica della milestone;
2. promozione di `TestClusterConvergence` a **test Compose reale** basato sul deployment `docker-compose.yml` di root;
3. aggiunta di `tests/integration/compose_harness_test.go` per orchestrare gli script canonici `cluster_up`, `cluster_wait_ready`, `cluster_down` e la raccolta artefatti;
4. mantenimento della variante veloce `TestClusterConvergenceInMemory` per debugging locale senza Docker;
5. allineamento dei file Compose/config (`configs/node1.yaml`, `configs/node2.yaml`, `configs/node3.yaml`) ai parametri M09 reali con aggregazione `average` e `initial_value` rispettivamente `10`, `30`, `50`;
6. aggiornamento della documentazione canonica (`docs/testing.md`) per distinguere in modo netto suite reale e suite veloce.

## Strategia test

La strategia canonica M09 usa ora un cluster reale multi-nodo locale:

- cluster a `3` nodi (`node-1`, `node-2`, `node-3`) avviati come servizi Compose `node1`, `node2`, `node3`;
- deployment reale tramite `docker-compose.yml` di root e script `scripts/cluster_up.sh`, `scripts/cluster_wait_ready.sh`, `scripts/cluster_down.sh`;
- aggregazione attiva `average` su tutti i nodi;
- valori iniziali `10`, `30`, `50` impostati nei file config runtime con il nuovo parametro `initial_value`;
- readiness osservabile tramite stato `running` dei container e marker di log `gossip bootstrap completato` + `transport gossip avviato`;
- raccolta dei valori finali per nodo dai log strutturati di shutdown, poi riformattati nello stesso report `formatNodeObservation(...)`.

Per non perdere la suite rapida storica, il repository conserva anche una variante separata `TestClusterConvergenceInMemory`, ma questa non è più il riferimento canonico della milestone.

## Criterio di convergenza

Il criterio di convergenza adottato è esplicito e osservabile:

- `max(values) - min(values) <= 0.05`

Dove `values` è il vettore dei valori aggregati correnti dei nodi del cluster. Il valore atteso comune dello scenario resta `30.0`, cioè `average(10, 30, 50)`, ed è usato nel report finale per calcolare la differenza di ciascun nodo dal riferimento. Il pass/fail resta comunque determinato dalla banda massima tra nodi.

## Timeout motivati

I timeout di M09 sono ora sdoppiati fra test canonico reale e variante veloce:

- `composeReadyTimeout = 90s`;
- `composeConvergenceTimeout = 18s`;
- `composeShutdownTimeout = 40s`;
- `m09InMemoryTimeout = 350ms`.

Motivazioni:

- il cluster reale richiede margine per build immagini, avvio container, bootstrap gossip reale e teardown pulito con artefatti finali;
- la finestra di convergenza Compose da `18s` copre più round reali con `gossip_interval_ms = 1000`;
- la suite in-memory conserva il timeout breve storico per mantenere un entrypoint rapido di regressione locale.

## Formato report finale

Il report finale per nodo è stato stabilito nel formato:

```text
node_id=<id> observed_value=<valore> expected_delta=<differenza_dal_valore_atteso> common_band=<banda_cluster>
```

Questo formato rende immediatamente leggibili sia il valore osservato dal singolo nodo sia la distanza dal riferimento o, alternativamente, la banda comune condivisa dal cluster nello stesso snapshot finale.

## Limiti/rischi

- Il test canonico dipende dalla disponibilità di Docker/Compose nell'ambiente di esecuzione.
- La convergenza viene verificata tramite un cluster locale reale, ma resta limitata a tre nodi e a un deployment Compose singolo host.
- La variante in-memory continua a essere utile perché la suite Compose è necessariamente più lenta e più costosa.
