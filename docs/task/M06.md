## Scelte fatte sulla strategia config
La strategia di configurazione reale implementata è la seguente.

### 1. Precedence esplicita e deterministica
Il flusso di caricamento resta:

```text
Default() -> parse file config -> overrideFromEnv() -> Validate()
```

Questa scelta rende prevedibile il comportamento runtime e consente di usare:
- default sensati per sviluppo locale;
- file esterni come base configurabile;
- env override per deployment e test;
- una validazione finale bloccante prima dell'avvio del nodo.

### 2. Parser YAML volutamente minimale
Il loader YAML non usa un parser generico: supporta solo il sottoinsieme davvero richiesto dal runtime corrente, cioè:
- coppie `chiave: valore` top-level;
- liste inline `[a,b,c]`;
- liste multilinea solo per `bootstrap_peers`, `seed_peers`, `enabled_aggregations`.

La scelta riduce complessità e ambiguità del contratto attuale, ma restringe la sintassi accettata rispetto a YAML completo.

### 3. Discovery peer con preferenza per `bootstrap_peers`
La funzione `DiscoveryPeers()` privilegia `bootstrap_peers`; se la lista è vuota usa `seed_peers` come fallback. In `cmd/node/main.go` questo risultato viene passato al bootstrap membership insieme a `join_endpoint`.

Quindi la strategia effettiva è:
- `join_endpoint` opzionale come meccanismo di join iniziale;
- `bootstrap_peers` come lista di discovery preferita;
- `seed_peers` come fallback compatibile.

### 4. Override env tolleranti ma non silenziosamente correttivi
Gli override env per interi e CSV vengono applicati solo se il parsing riesce. Se l'env è malformato, il valore precedente non viene sostituito; la configurazione finale viene poi comunque sottoposta a `Validate`.

Questa scelta evita di corrompere la config con input env illegibili, ma non produce un errore immediato specifico sull'env malformato se il valore preesistente resta valido.

## Validazioni introdotte
Le validazioni effettivamente presenti in `Validate(cfg)` e coperte dalla suite test sono:

### Campi obbligatori e numerici
- `node_id` obbligatorio;
- `bind_address` obbligatorio e compatibile con `net.JoinHostPort`;
- `node_port` nell'intervallo `1..65535`;
- `gossip_interval_ms > 0`;
- `fanout > 0`;
- `membership_timeout_ms > 0`;
- `aggregation` obbligatoria;
- `enabled_aggregations` non vuota.

### Endpoint e peer list
- `join_endpoint`, se presente, deve rispettare `host:porta`;
- ogni elemento di `bootstrap_peers` e `seed_peers` deve avere formato valido `host:porta`;
- peer list con item vuoti vengono rifiutate;
- peer list con duplicati vengono rifiutate.

### Aggregazioni supportate
- il set supportato è esattamente `sum`, `average`, `min`, `max`;
- `enabled_aggregations` rifiuta valori vuoti, duplicati e aggregazioni non supportate;
- `aggregation` deve appartenere al set supportato;
- `aggregation` deve appartenere anche a `enabled_aggregations`.

### Caricamento file e parsing
- sono accettati solo file `.json`, `.yaml`, `.yml`;
- mismatch di tipo in JSON sono bloccanti;
- valori numerici non parseabili in YAML sono bloccanti;
- liste YAML malformate sono bloccanti.

## Eventuali limiti residui
Restano i seguenti limiti, coerenti con il comportamento reale attuale:
- il parser YAML è intenzionalmente minimale e non supporta YAML arbitrario o strutture annidate generiche;
- `fanout` è validato e documentato, ma nel runtime gossip corrente non risulta ancora applicato alla selezione dei peer del round;
- gli override env numerici/CSV malformati non generano un errore immediato dedicato se il valore precedente resta valido;
- `cmd/node/main.go` usa ora un client HTTP concreto verso `http://<join_endpoint>/join` quando `join_endpoint` è valorizzato; se il join fallisce oppure non è configurato, il bootstrap mantiene il fallback su `DiscoveryPeers()` (`bootstrap_peers`, poi `seed_peers`).
