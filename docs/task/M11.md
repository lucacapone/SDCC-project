# M11 — Documentazione operativa dell'observability minima

## Obiettivo
Documentare in modo tracciabile lo stato post-M11 dell'observability del repository, chiarendo la baseline iniziale rilevata, le modifiche documentali introdotte, i file toccati, le decisioni progettuali, i test eseguiti e i limiti/follow-up ancora presenti.

## Stato iniziale rilevato
All'inizio dell'attività la situazione osservata nel repository era la seguente:

- il runtime e il package `internal/observability` esponevano già una **API minima funzionante** con logger strutturato, collector metriche, endpoint `/health`, `/ready`, `/metrics` e test canonico `TestMetricsExposure`;
- `README.md`, `docs/architecture.md` e `docs/testing.md` contenevano già riferimenti sintetici all'observability minima e al comando di verifica mirato;
- **mancava però un documento dedicato `docs/observability.md`** che consolidasse in un solo punto architettura minima, campi log, metriche, endpoint, istruzioni d'uso, limiti noti e scelte progettuali;
- **mancava inoltre il task report `docs/task/M11.md`** nello stesso stile adottato per `docs/task/M08.md`, `docs/task/M09.md` e `docs/task/M10.md`;
- il `README.md` non dichiarava ancora in modo esplicito **lo stato post-M11** nella sezione milestone né il comando canonico di verifica M11 come riferimento primario della milestone.

## Cosa è stato implementato/modificato
Per chiudere M11 è stato eseguito un consolidamento documentale coerente con lo stato reale del repository:

1. **creazione di `docs/observability.md`** come documento canonico dedicato all'observability minima del nodo;
2. documentazione esplicita di:
   - architettura minima del sottosistema;
   - campi log supportati;
   - metriche esposte;
   - endpoint disponibili;
   - istruzioni operative d'uso e verifica;
   - limiti noti e scelte progettuali;
3. **creazione di `docs/task/M11.md`** nello stesso stile delle milestone documentali recenti;
4. **aggiornamento del `README.md`** per dichiarare lo stato post-M11 nella sezione milestone e nelle sezioni operative dedicate all'observability, indicando come riferimento canonico il comando:

```bash
go test ./internal/observability -run TestMetricsExposure
```

## File toccati
File coinvolti nella chiusura documentale della milestone M11:

- `docs/observability.md`;
- `docs/task/M11.md`;
- `README.md`;
- `docs/operational_log.md`.

## Decisioni progettuali
Le decisioni progettuali/documentali emerse in M11 sono le seguenti:

1. **M11 viene chiusa come milestone di consolidamento documentale**, non di estensione runtime, perché l'API minima di observability era già presente nel repository.
2. **`docs/observability.md` diventa il punto di riferimento dedicato** per uso operativo e verifica rapida, mentre `README.md` resta introduttivo/sintetico.
3. **Il comando canonico di verifica viene mantenuto unico e minimale**: `go test ./internal/observability -run TestMetricsExposure`.
4. **La documentazione resta aderente all'implementazione reale**: nessun riferimento a tracing distribuito, exporter esterni o stack di monitoring non presenti nel repository.
5. **La separazione health/readiness/metrics viene esplicitata chiaramente** per evitare ambiguità operative nei controlli manuali o in Compose.

## Test eseguiti
Le verifiche rilevanti per M11 sono le seguenti:

```bash
go test ./internal/observability -run TestMetricsExposure
```

Questa verifica resta la prova canonica della milestone perché congela l'esposizione minima delle metriche e la coerenza base degli endpoint di observability.

## Limiti e follow-up
Limiti residui:

- la milestone M11 non introduce una piattaforma observability completa, ma solo la sua documentazione canonica minima;
- il comando di verifica resta focalizzato sul package `internal/observability` e non copre da solo un intero cluster Compose o un deployment AWS-like;
- le metriche restano intenzionalmente a bassa cardinalità e quindi non sostituiscono strumenti di tracing/debug distribuito più ricchi.

Follow-up consigliati:

1. collegare in futuro `docs/observability.md` a eventuali playbook operativi Compose/AWS se verranno introdotti controlli runtime più ricchi;
2. estendere la documentazione se il formato `/metrics` o i campi log evolveranno oltre il perimetro minimo attuale;
3. mantenere aggiornati `README.md`, `docs/architecture.md`, `docs/testing.md` e `docs/observability.md` quando cambia la semantica degli endpoint o dello stato lifecycle.
