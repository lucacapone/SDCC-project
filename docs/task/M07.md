# M07 — Compose + networking

## Obiettivo
Documentare in modo strutturato la milestone M07 relativa al deployment locale multi-nodo con Docker Compose, chiarendo lo stato iniziale rilevato, i gap rispetto ai done criteria, le modifiche introdotte negli artefatti di deployment e il risultato atteso/verificabile del flusso `docker compose up -d --build && docker compose ps`.

## Stato iniziale rilevato
Prima del consolidamento M07 il repository presentava i seguenti elementi e limiti iniziali:
- alla root era presente un `docker-compose.yml` placeholder, non allineato al cluster locale realmente descritto o atteso per la milestone;
- il Compose multi-nodo reale era collocato in `deploy/docker-compose.yml`, creando ambiguità sulla sorgente operativa corretta;
- mancava un `Dockerfile` applicativo di root per costruire un'immagine locale dedicata e riusabile dai servizi Compose;
- mancava una documentazione di deployment locale esplicita e canonica (`docs/deployment.md`) che descrivesse prerequisiti, topologia, troubleshooting e procedura operativa standard.

Questa situazione rendeva possibile avviare il progetto solo con conoscenza implicita dello stato del repository o usando path/istruzioni non univoche.

## Gap rispetto ai done criteria
Rispetto ai done criteria dichiarati per M07, i gap iniziali erano i seguenti:

1. **Almeno 3 nodi avviati e raggiungibili**
   - il file Compose operativo non era chiaramente identificato come sorgente canonica;
   - il placeholder di root non garantiva un cluster locale multi-nodo realmente usabile.

2. **Bootstrap/membership funzionanti su rete Compose**
   - l'ambiguità tra file Compose di root e file in `deploy/` aumentava il rischio di usare hostname o parametri non coerenti;
   - senza una rete Compose dedicata documentata, il ruolo dei service name come endpoint DNS non era esplicitato in modo operativo.

3. **Comandi start/stop/documentazione allineati**
   - mancava una pagina `docs/deployment.md` come riferimento canonico;
   - README e file Compose non erano ancora organizzati in modo da indicare senza dubbio il flusso standard `docker compose ...` dalla root.

## Modifiche introdotte
Le modifiche introdotte per chiudere M07 sono le seguenti.

### 1. `docker-compose.yml`
- il file Compose di root è stato promosso a sorgente canonica del cluster locale;
- è stata definita una topologia a tre servizi (`node1`, `node2`, `node3`) basata su un anchor comune `x-node-base`;
- tutti i servizi usano la stessa immagine locale `sdcc-node:local` costruita dal `Dockerfile` di root;
- ogni nodo monta il proprio file `configs/node*.yaml` come `/config/config.yaml` e avvia il binario con `--config /config/config.yaml`;
- i servizi sono collegati alla rete bridge dedicata `sdcc-net`.

### 2. `Dockerfile`
- è stato introdotto un `Dockerfile` di root per costruire l'immagine applicativa locale invece di usare container ad-hoc con `go run`;
- la build dell'applicazione è stata resa parte del flusso Compose standard, così che `docker compose up -d --build` sia sufficiente per generare ed eseguire il cluster.

### 3. `docs/deployment.md`
- è stato creato il documento canonico di deployment locale multi-nodo;
- sono stati documentati prerequisiti, porte, topologia del cluster, rete Compose, uso dei service name come DNS interno, troubleshooting e procedura operativa minima;
- è stata esplicitata la distinzione tra file Compose canonico di root e file storico/non operativo in `deploy/`.

### 4. `README.md`
- il README è stato allineato al nuovo flusso operativo standard basato sul `docker-compose.yml` di root;
- sono stati aggiunti i comandi reali di avvio, verifica, log e stop (`docker compose up -d --build`, `docker compose ps`, `docker compose logs -f ...`, `docker compose down`);
- è stato chiarito che la scoperta dei peer nel cluster locale usa i service name Compose `node1`, `node2`, `node3` tramite DNS interno della rete dedicata.

## Scelte progettuali
Le scelte progettuali adottate per M07 sono state le seguenti.

### Compose canonico alla root
La scelta principale è stata usare `docker-compose.yml` alla root come unico entry point operativo del cluster locale. Questo riduce ambiguità, semplifica i comandi e allinea la documentazione ai comportamenti reali attesi da chi clona il repository.

### Immagine applicativa locale condivisa
Tutti i servizi Compose riusano la stessa immagine locale `sdcc-node:local`, costruita dal `Dockerfile` del repository. In questo modo:
- si evita la duplicazione della logica di avvio nei servizi;
- si rende riproducibile la build dell'applicazione;
- si separa in modo più pulito build dell'immagine e configurazione runtime dei nodi.

### Rete dedicata Compose
È stata adottata una rete bridge dedicata `sdcc-net` per isolare e rendere riproducibile la comunicazione del cluster locale. Questa scelta:
- evita dipendenze da reti Docker implicite o condivise con altri stack;
- rende esplicita la topologia locale del deployment;
- facilita il troubleshooting dei problemi di connettività tra nodi.

### Uso dei service name come DNS
I peer di bootstrap/configurazione usano i service name Compose (`node1`, `node2`, `node3`) come hostname DNS interni. Questa scelta è importante perché:
- Docker Compose risolve automaticamente i service name sulla rete dedicata;
- evita di dipendere da IP dinamici dei container;
- mantiene allineati file config, Compose e comportamento runtime.

## Limiti residui ed edge case noti
Restano alcuni limiti o edge case coerenti con la natura del deployment locale multi-container:

- **race condition al bootstrap**: i container possono partire quasi simultaneamente e nei primi secondi alcuni peer potrebbero non essere ancora pronti;
- **differenze tra ambienti Docker locali**: Docker Desktop, Engine Linux nativo o ambienti virtualizzati possono mostrare comportamenti leggermente diversi lato networking o tempi di startup;
- **disallineamento YAML/env**: il runtime supporta override environment, quindi valori incoerenti tra `configs/node*.yaml` e variabili ambiente Compose possono produrre diagnosi fuorvianti;
- **assenza di prova automatica forte del bootstrap nel task documentale**: il comando `docker compose ps` verifica in modo immediato lo stato dei container, ma non certifica da solo l'effettiva convergenza gossip interna senza ispezione dei log;
- **dipendenza dai nomi DNS Compose**: se si rinominano i servizi senza aggiornare `seed_peers`, la discovery locale fallisce o diventa parziale.

## Risultato finale atteso / verificato
Il risultato finale atteso per M07 è il seguente:
- dalla root della repository il comando `docker compose up -d --build` costruisce l'immagine locale e avvia tre nodi del cluster (`node1`, `node2`, `node3`);
- il comando successivo `docker compose ps` mostra i servizi Compose del cluster in stato attivo;
- i nodi usano la rete dedicata `sdcc-net` e i service name Compose come endpoint DNS per i peer dichiarati in configurazione;
- documentazione e file operativi puntano tutti allo stesso flusso standard.

Come criterio operativo della milestone, la verifica minima resta:

```bash
docker compose up -d --build && docker compose ps
```

Questa verifica conferma che il deployment locale multi-nodo è predisposto in modo coerente con la struttura attuale del repository e con i done criteria documentali della milestone M07.

## Artefatto di verifica ripetibile introdotto
Per rendere ripetibile la verifica M07 senza dipendere da passaggi manuali è stato aggiunto lo script:

```bash
scripts/m07_collect_compose_evidence.sh
```

Lo script:

- esegue dalla root `docker compose up -d --build`;
- attende pochi secondi per permettere bootstrap e primi round gossip;
- salva `docker compose ps`;
- salva una raccolta minima dei log di `node1`, `node2`, `node3`;
- archivia tutto in `artifacts/m07/<timestamp>/` e aggiorna il link `artifacts/m07/latest`.

### Output atteso
Una singola esecuzione deve produrre almeno i file seguenti:

- `compose-up.txt`;
- `compose-ps.txt`;
- `node1.log`;
- `node2.log`;
- `node3.log`;
- `summary.txt`.

### Criteri osservabili nei log
I criteri minimi osservabili restano coerenti con la documentazione canonica di deployment:

1. **bootstrap completato** → presenza di `gossip bootstrap completato` con `event=node_bootstrap`;
2. **peer discovery via service name Compose** → presenza nei campi log di endpoint `node1:7001`, `node2:7002`, `node3:7003`;
3. **membership iniziale non vuota o convergente** → `peers>0` nei log iniziali oppure presenza di `event=remote_merge`, `event=gossip_round`, `membership_entries=<n>` nei round successivi.

### Comando operativo consigliato
```bash
./scripts/m07_collect_compose_evidence.sh
```
